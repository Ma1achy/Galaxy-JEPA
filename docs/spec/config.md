# Spec — config + provenance

*Status: design proposal for sign-off. Expands `docs/architecture.md` → "Config is the
experiment record". Implemented (Track B) in `src/galaxy_jepa/core/config.py`. British
English.*

A run is **fully determined by** `(config hash, code SHA, data-snapshot version, seed,
escape_hatches_used)`. Every artefact is stamped with that tuple, so the three headline
figures regenerate from their provenance and no figure can hide a methodological
shortcut. The config *is* the experiment — no runtime config fetching, no mid-run
reloading.

---

## 1. Config derivation — **decided: auto-capture from `__init__`**

A component derives its config from its own constructor signature; there is no separate,
driftable schema to keep in sync ("configuration over runtime cleverness"). The base is
`Configurable`:

- `__init_subclass__` introspects each subclass's `__init__` and **rejects
  `*args`/`**kwargs`** at class-definition time — a component must have a fixed,
  explicit signature to be reproducible by construction.
- Constructor kwargs are **captured at instantiation** (a wrapped `__init__`), then
  `to_config()` / `from_config()` round-trip the component.

### Nested configs + class references (built now, not deferred)

A real run config nests sub-configs (model inside objective inside run) and carries
class references on day one — neither JSON-serialises naively. So serialisation is
**recursive** and a class travels as a **`ClassRef`** — a `"module:qualname"` string
resolved back with `importlib`:

```python
to_config()  ->  {"__class__": "pkg.mod:Outer",
                  "config": {"inner": {"__class__": "...:Inner", "config": {...}},
                             "tag":   {"__classref__": "pkg.mod:SomeClass"}}}
```

`_serialise` handles `Configurable`, class objects (`ClassRef`), pydantic models,
primitives, and containers; **anything else raises** — a config is serialisable by
construction, never silently stringified (fail loudly).

`config_hash` = `sha256` over the **canonical JSON** (sorted keys) of the recursively
serialised tree — so a hash is stable across a `to_config → from_config → to_config`
round-trip and changes iff a value changes.

## 2. Typed top-level run config

`RunConfig(pydantic.BaseModel)` is the base for the YAML-loaded run config, validated at
load with `extra='forbid'` — an unknown key is a loud error, not a silently-ignored typo.
(`configs/pretrain.yaml`, `configs/probe.yaml` become `RunConfig` subclasses when the
loop lands.)

---

## 3. The run-stamp

`RunStamp` (frozen dataclass): `config_hash`, `code_sha`, `code_dirty`, `data_snapshot`,
`seed`, `escape_hatches_used`.

- **`code_sha`** — `git rev-parse HEAD` + a dirty-tree flag (`git status --porcelain`).
  Outside a git repo it records the sentinel `"nogit"` with a loud warning and
  `code_dirty=True` — a run may proceed un-versioned, but its irreproducibility is
  recorded, never hidden.
- **`data_snapshot`** — supplied by the data layer as a **manifest hash** (object IDs +
  the pull query), so reproducibility is structural rather than a hand-bumped version
  string someone must remember to increment.
- **`escape_hatches_used`** — the list of forfeited-guarantee identifiers (see
  `docs/spec/escape-hatches.md`).
- **Stretch parameters** (asinh `Q` + per-channel flux scale; `docs/spec/data.md`) are
  ordinary config and therefore enter the config hash automatically — the stretch is a
  stamped, reproducible decision, never a notebook constant.

---

## 4. Artefact layout + figure regeneration

`write_stamp(stamp, artefact_dir, config)` writes `stamp.json` (and the full
`config.json`) into `artefacts/<run-id>/`. A figure is regenerated by reading its
`config.json`, reconstructing the components via `from_config`, and re-running with the
stamped `seed` — the scientific analogue of artefact atomicity.
